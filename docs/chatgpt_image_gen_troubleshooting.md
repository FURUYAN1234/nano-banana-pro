# ChatGPT 画像生成 4コマ漫画 — 事故防止プロトコル & 対策

> 最終更新: 2026-05-08
> 対象: ChatGPT (GPT-5.5 Thinking) ブラウザ版の画像生成機能
> 状態: **正式ドキュメント**

---

## 1. 発生した事故の概要

2026-05-07、モバイル回線＋別PCからChatGPTで4コマ漫画を生成した際、以下の事故が同時発生した。

| # | 事故 | 深刻度 |
|---|------|--------|
| 1 | セリフが全部**横書き**になった | 🔴 致命的 |
| 2 | シナリオにない**キャラが勝手に台詞**を話した | 🔴 致命的 |
| 3 | 構図が全部**アイレベル**に。カメラワーク・レンズ効果消失 | 🔴 致命的 |
| 4 | キャラシート＋プロンプト同時添付時、**1回目にキャラシートがそのまま再現**された | 🟠 重大 |

---

## 2. 原因分析 — ChatGPTの画像生成の内部構造に起因する問題

### 2.1 ChatGPTの画像生成パイプラインの特性

ChatGPTの回答から判明した重要な事実:

```
長文をそのまま全部忠実に処理するというより、
内部で要約・整理されてから画像生成に渡る挙動になりやすい
```

**Gemini/Imagen APIとの決定的な違い**:
Nano Banana ProのGemini APIは、プロンプトをそのままテキストとして画像生成モデルに渡す。
一方ChatGPTは、LLM層が介在し、ユーザーの長文プロンプトを**内部で要約・再解釈**してから画像生成ツールに渡す。
この「中間処理」で条件が落ちる。

### 2.2 各事故の個別原因

#### 事故1: セリフの横書き化

- ChatGPTの画像生成側が**英語圏デフォルト**の横書きテキスト描画に寄りやすい
- 長文プロンプトの中で「縦書き」指示が他の条件に埋没し、優先度が下がった
- Nano Banana Pro KI対応: `Compact Text Density (v1.8.32)` で「Vertical Flow」を明示しているが、ChatGPTでは不十分

#### 事故2: セリフの勝手追加

- ChatGPTが「漫画らしさ」を補完するため、指定外の擬音・ツッコミ・ナレーションを自動生成
- **「漫画」「議論」「図書室」**等のキーワードが「文字が多いべき」と解釈された
- Nano Banana Pro KI対応: `SFX ARE PHYSICAL 3D OBJECTS (v1.8.30)` は擬音制御だがセリフ追加防止は別問題

#### 事故3: カメラワーク平易化

- 画像生成側が**可読性・キャラの顔・5人配置を優先**し、カメラ指定を犠牲にした
- 長文内で俯瞰・ローアングル・ダッチアングル・魚眼が「安全側」に丸められた
- Nano Banana Pro KI対応: `ABSOLUTE PHYSICAL GEOMETRY LOCK` と `Camera Force (Anti-Eye-Level)` が存在するが、ChatGPTのLLM中間処理で圧縮される

#### 事故4: 参照画像のキャラシート再現

- ChatGPTが添付画像を「キャラデザイン参照」ではなく「画面に反映すべきビジュアル資料」として解釈
- 特にキャラクターシート形式＋長文＋「reference」「exact」系の強い命令の組み合わせで発生
- **Nano Banana Proにはこの問題への対策が存在しない**（Gemini APIは添付画像参照の仕組みが異なるため）

### 2.3 通信環境の影響

```
通信環境は関係し得る。
ただし「通信が悪いからキャラシートが出る」ではなく、
通信不安定がきっかけで参照画像・長文プロンプトの受け渡しや生成処理が失敗寄りになる
```

影響する要因:
- モバイル回線（テザリング等）
- VPN / プロキシ
- ブラウザ拡張機能
- セキュリティソフト
- キャリア回線のパケット最適化

## 3. 対策 — 3段階アプローチ

### 3.1 【構造戦略】プロンプト圧縮＆優先順位の明示

**核心**: ChatGPTの画像生成では「全部の命令が同じ強度で届かない」。
したがって、**絶対に落とされたくない条件を冒頭に集約**し、装飾的な詳細は後半に置く。

#### A. ChatGPT向け「プロンプト階層」

```
┌─────────────────────────────────────┐
│  Layer 0: 生成事故防止プロトコル      │  ← 冒頭。最高優先度
│  (参照画像誤用禁止/横書き禁止/       │
│   セリフ追加禁止/カメラ維持)          │
├─────────────────────────────────────┤
│  Layer 1: フォーマット＆タイトル      │  ← A4縦, 4等分, 右→左
├─────────────────────────────────────┤
│  Layer 2: キャラ表（圧縮版）          │  ← 1キャラ1行。タグ不要
├─────────────────────────────────────┤
│  Layer 3: 各パネル                    │  ← 構図・行動・セリフのみ
│  (1パネル3〜5行に絞る)               │     装飾タグ削除
├─────────────────────────────────────┤
│  Layer 4: 最終チェックリスト          │  ← 末尾で再確認を強制
└─────────────────────────────────────┘
```

#### B. キャラ表の圧縮

**推奨**: 1キャラ1行の短縮形式

```
アカリ＝橙ボブ・裸眼
ヒカリ＝金髪ストレートボブ・丸眼鏡
ミク＝褐色・金髪ウェーブ・ピンクメッシュ・裸眼
リン＝茶髪ツインテ・眼鏡
サエコ＝黒髪ロング・裸眼
```

**注意**: Nano Banana Proの `--no fused characters, glasses on non-glasses characters` 等のネガティブプロンプトは、
**ChatGPTでは逆効果**の可能性がある。ChatGPTの画像生成はネガティブプロンプト構文を理解しない。

#### C. パネル記述の簡素化

**各パネル3〜5行ルール**:

```
Panel 1 (起):
Camera: 俯瞰
Layout: アカリ右→ヒカリ中央→ミク左
Action: アカリが本を叩きつける
Dialogue:
  アカリ「今日の議題は——！」
```

`(14mm Wide Angle Lens:1.5)` のような重み付き構文はChatGPTでは無効なので**全削除**する。

### 3.2 【プロンプトテンプレート】事故防止ブロック

#### 核心版（短縮・最優先5行）— 冒頭・各パネル前・末尾に3箇所配置

```text
HIGHEST PRIORITY / 最優先:
1. 指定セリフ以外の文字を絶対に追加しない / No text except specified Dialogue
2. すべてのセリフは縦書き日本語 / All dialogue vertical Japanese
3. 添付画像はキャラ識別専用。キャラシート構図を再現しない / Reference = identity only
4. カメラワークを平凡な目線構図に戻さない / Do NOT flatten camera work
5. 最終画像は新規4コマ漫画シーン / Output = new 4-panel manga scene
```

#### 完全版ブロック（冒頭配置）

ChatGPTが提案した `[CRITICAL MASTER CONTROL BLOCK — ABSOLUTE TOP PRIORITY]` をそのまま使用する。

**追加すべきNano Banana Pro固有の知見:**

- `RIGHT/CENTER/LEFT` ゾーンスロッティング (v1.8.39) → 各パネルのキャラ配置を明示
- `Location Text Ban` (v1.8.71) → 背景の場所名をテキストとして描かない
- `Speaker-on-Right Rule` (v1.8.38) → 最初の話者を右側に配置

#### 末尾チェックリスト（出力前検証用）

```text
[FINAL CHECK — REDRAW IF ANY FAILS]

✅ 全セリフが縦書き日本語か？
✅ 横書きセリフは0か？
✅ Dialogue指定外のセリフを追加していないか？
✅ 指定外の擬音・謎文字・英語を描いていないか？
✅ キャラシート風レイアウトを再現していないか？
✅ 白背景の立ち絵・表情一覧になっていないか？
✅ 4コマ漫画の物語シーンになっているか？
✅ カメラワークが平凡な目線構図に戻っていないか？
✅ 参照画像はキャラ識別だけに使われているか？
✅ 眼鏡の有無は全キャラ正しいか？
```

### 3.3 【環境対策】通信・端末チェックリスト

| # | チェック項目 | 方法 |
|---|------------|------|
| 1 | 固定光回線で同じプロンプトを試す | 自宅PCで再現テスト |
| 2 | VPN/プロキシを切る | 接続設定確認 |
| 3 | ブラウザを変える | Chrome→Edge or Firefox |
| 4 | シークレットウィンドウで試す | 拡張機能の影響排除 |
| 5 | 添付画像を再アップロード | 画像キャッシュのリセット |
| 6 | 短いプロンプトでまずテスト | 「キャラシート構図を出すな」が効くか確認 |

## 4. ChatGPT vs Gemini/Imagen — プロンプト処理の違い

| 項目 | Nano Banana Pro (Gemini) | ChatGPT ブラウザ版 |
|------|-------------------------|-------------------|
| プロンプト処理 | テキストを直接画像モデルへ渡す | LLMが内部で要約→画像生成ツールへ |
| 長文耐性 | 比較的高い（APIの上限まで） | **低い**（内部圧縮で条件が落ちる） |
| ネガティブプロンプト | `--no` 構文で制御可能 | **非対応**（逆効果の可能性） |
| 重み付き構文 | `(tag:1.5)` で効果あり | **無効**（通常テキスト扱い） |
| 参照画像 | APIで別途送信・制御可能 | 添付画像の解釈が**不安定** |
| キャラシート混入 | 発生しにくい | **発生しやすい** |
| 縦書き/横書き | 制御が比較的安定 | **不安定**（横書きデフォルト） |
| カメラワーク | 極端な指定が通りやすい | **平易化されやすい** |

### 統合方針

1. **Gemini API側**: 既存KI (v1.8.89) のプロンプトエンジニアリングはそのまま維持
2. **ChatGPT側**: 本ドキュメントの対策を適用。技術タグは自然言語に全面置換
3. **共通**: キャラ識別情報は共有可能。ChatGPT向けは1キャラ1行の圧縮形式

## 6. まとめ

ChatGPTの画像生成は、Gemini/Imagen APIとは根本的に異なる内部処理パイプラインを持つ。

FURUさんのプロンプトが「プロ仕様で制御項目が多い」ことは強みだが、ChatGPTでは
**「詳細すぎて逆に解像度が下がる」ゾーンに入っている**（ChatGPT側の分析）。

**対策の核心:**
1. **冒頭に絶対条件を集約** — 長文の中で埋没させない
2. **装飾タグを削除し自然言語に** — ChatGPTは `(tag:1.5)` や `--no` を理解しない
3. **冒頭・パネル前・末尾の3箇所に核心5行を重複配置** — 条件脱落を防ぐ
4. **参照画像の「形式再現禁止」を明示** — Geminiにはない問題への新規対策
5. **通信環境を安定させる** — 固定回線＋PC＋VPNなしが最安定
